Database Web UI (Angular)

GraphLib includes an optional local-only Angular web UI for inspecting and editing the SQLite database. This UI is intended as a living manual and operational tool, not a production service.

What this UI is

  • A local Angular application
  • Used to browse, edit, and visualize data in GraphLib.db
  • Focused on human understanding, not automation
  • A companion to the CLI, not a replacement

What this UI is not

  • Not a hosted web site
  • Not a production admin portal
  • Not another .NET web app
  • Not required to run GraphLib
Design note: The Angular UI intentionally stays “funky.” It is a developer-facing tool, not an enterprise platform.

Database requirement

The Angular Database UI only works if a GraphLib SQLite database already exists. The UI does not create or initialize databases.

By convention, GraphLib databases usually live at one of the following paths:


src\GraphLib.Console\bin\Debug\net8.0\Data\GraphLib.db
src\GraphLib.Console\bin\Release\net8.0\Data\GraphLib.db

If no database is found, the UI should clearly indicate that state and disable all database-related navigation.

Intended capabilities

Database navigation

  • List tables
  • Browse table contents
  • Filter, sort, and page through rows (DataTables)
  • Edit rows via a safe UI (modal-based editing preferred)

Editing philosophy

The UI should support CRUD operations without requiring users to write SQL. SQL is considered an advanced capability and should be optional.

  • Editing via forms / modals
  • Primary-key based updates
  • No assumption that users understand SQLite schema details

SQL query runner (advanced)

A simple SQL query runner may be provided for power users. This is primarily for inspection and debugging.

  • SELECT queries by default
  • Write queries only if explicitly enabled
  • Results displayed in tabular form

Visualization

The UI may use Chart.js to visualize database data, especially operational metadata.

Examples include:

  • File extension counts
  • Conversion success vs failure
  • Most common source → target conversions

Visualizations are exploratory and explanatory, not meant for compliance dashboards.

Relationship to the CLI

The GraphLib CLI remains the primary execution surface. The Angular UI exists to:

  • Explain what the CLI produced
  • Make the database approachable
  • Serve as living documentation

If the UI and CLI ever disagree, the CLI is considered the source of truth.

Local-only by design

This UI is intentionally designed to run locally:

  • No authentication
  • No multi-user assumptions
  • No deployment pipeline
  • No hosting requirements

If it ever becomes more than that, it will be treated as a separate product.
